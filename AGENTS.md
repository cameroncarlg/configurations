# configurations — Nix flake for macOS + NixOS

Act like a coding agent here: read files, propose edits, run checks.

## Readme

A unified Nix configuration managing macOS and Linux environments through a
single flake, providing consistent development environments with
platform-specific customizations.

- **Cross-platform**: macOS (Apple Silicon) via nix-darwin, Linux via NixOS
- **System**: Fish shell, GNOME (Linux), VPN support
- **Home**: Consistent user environment via home-manager (as a module)
- **Server**: Media stack (Jellyfin, Sonarr), Mealie, Mullvad VPN, Minecraft

See **Rebuild commands** below for usage.

## What this is

A single flake managing two machines:

| Host | System | Attr | Entry |
|---|---|---|---|
| `Camerons-MacBook-Pro` | `aarch64-darwin` (nix-darwin) | `darwinConfigurations."Camerons-MacBook-Pro"` | `hosts/macbook/` |
| `nixos` (home server) | `x86_64-linux` (NixOS) | `nixosConfigurations.nixos` | `hosts/nixos-server/` |

Both use **home-manager as a module** (not standalone) with user `cameron`.

## Layout

```
flake.nix                     # inputs + both host outputs
hosts/
  macbook/
    configuration.nix         # darwin system config (small, ~33 lines)
    home.nix                  # home-manager for mac
  nixos-server/
    configuration.nix         # NixOS system config (LARGE, ~1200 lines — media stack, minecraft, etc.)
    hardware-configuration.nix  # host-generated; DO NOT edit by hand
    home.nix                  # home-manager for server
modules/
  common/                     # shared across both hosts (git, helix, fzf, ripgrep, lazygit, direnv, pi-agent)
  darwin/                     # mac-only (fish, nushell, starship, wezterm)
  linux/                      # linux-only (fish, nushell, starship, wezterm, minecraft, discord-bot)
```

Every `modules/<dir>/default.nix` is a barrel file — to add a new module,
create `foo.nix` in the right subdir AND add `./foo.nix` to that `default.nix`.

## Rebuild commands

```bash
# macOS
darwin-rebuild switch --flake ~/configurations

# NixOS server (run on the server itself)
sudo nixos-rebuild switch --flake ~/configurations#nixos

# Dry-run / test without activating
darwin-rebuild build --flake ~/configurations
nixos-rebuild build --flake ~/configurations#nixos

# Check flake evaluates (fast, no build)
nix flake check ~/configurations --no-build
```

**Before suggesting a rebuild, always try `nix flake check --no-build` first
so we catch eval errors cheaply.**

## Inputs

Pinned in `flake.nix`, passed through `specialArgs = { inherit inputs; }`
(system) and `home-manager.extraSpecialArgs = { inherit inputs; }` (HM).
To use an input inside a module, take `inputs` in the module's arg set:
`{ inputs, pkgs, ... }:`.

Current inputs: `nixpkgs` (unstable), `nix-darwin`, `home-manager`, `helix`,
`wezterm`, `nix-minecraft`, `llm-agents`.

Update the lock: `nix flake update` (whole lock) or
`nix flake lock --update-input <name>` (single input).

## Conventions

- **2-space indent**, trailing newline, no tabs.
- Module files are functions: `{ pkgs, ... }: { ... }` or
  `{ inputs, config, pkgs, ... }: { ... }` — only destructure what you use.
- Prefer adding to an existing module over creating new ones unless it's a
  clearly separate concern.
- Home-manager options go in `home.nix` or a `modules/common/*.nix` imported
  by it. System options go in `configuration.nix` or a `modules/<platform>/*.nix`.
- Don't mix: home-manager options in a NixOS module will eval-fail.

## Hard rules

- **Never edit `hosts/nixos-server/hardware-configuration.nix`** — regenerated by
  `nixos-generate-config`.
- **Never bump `home.stateVersion` or `system.stateVersion`** without me
  explicitly asking. They pin migration behavior.
- **Never commit secrets.** This repo has no sops/agenix yet; if a change needs
  a secret, stop and tell me — we'll wire up secrets management first. Real
  secrets live in `~/ssl`, `~/wireguard-keys`, or external stores.
- Don't run `nixos-rebuild switch` yourself. Propose the command; I'll run it.
  `darwin-rebuild build` and `nix flake check` are fine to run.
- Don't `git push`, `git commit`, or change branches without being asked.
  Current branch is often a feature branch (e.g. `working-nix-minecraft`).

## Gotchas

- `nix.enable = false` on macOS — Determinate Systems manages the Nix daemon.
  Don't "fix" this.
- Unfree packages need `nixpkgs.config.allowUnfree = true` or an explicit
  `allowUnfreePredicate`. If a package fails with "unfree", that's usually why.
- The server's `configuration.nix` is ~1200 lines and mixes networking, media
  services, homepage-dashboard, and more. When editing, read the surrounding
  section carefully — lots of commented-out experimental blocks.
- `modules/common/pi-agent.nix` writes `~/.pi/agent/AGENTS.md` via
  `home.file`. If that path already exists as a plain file, home-manager will
  refuse to activate — remove the stray file first.

## Quick checks before proposing a change

1. `nix flake check ~/configurations --no-build` — eval-clean?
2. `darwin-rebuild build --flake ~/configurations` or
   `nixos-rebuild build --flake ~/configurations#nixos` — builds?
3. Diff is minimal and scoped to one concern.
